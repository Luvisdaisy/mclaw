# ModWeaver 技术架构设计

## 文档信息
- **版本**：1.0
- **日期**：2026-06-01
- **作者**：架构团队
- **适用范围**：ModWeaver V2.5 产品实现

---

## 1. 架构概述

ModWeaver V2.5 是一个面向 Minecraft 模组管理的多 Agent 运行时编排平台。其核心设计目标为：

- **多智能体协作**：通过专业化 Agent（Planner、Executor、Diagnoser、Repairer、Memory、Evaluator）以 DAG 工作流模式协作完成复杂任务。
- **可观测与自进化**：内置量化评估引擎与反思记忆系统，使系统能从失败中自我改进。
- **资源治理与隔离**：提供断路器、限流、超时等运行时保护机制，并通过沙箱工作区实现执行环境完全隔离。
- **桌面应用形态**：基于 Electron + React 的跨平台桌面客户端，后端使用 FastAPI 提供高性能异步服务。

整体架构采用**分层+事件驱动**模式，将 GUI 表现层、Agent 运行时核心、工具执行层、沙箱隔离层解耦。Agent 之间通过消息总线（AgentBus）通信，工作流引擎负责编排并驱动 DAG 执行，资源治理层以切面方式拦截所有工具调用和事件流。

## 2. 逻辑架构视图

逻辑架构划分为以下层次和核心组件：

### 2.1 表现层
- **Electron Shell**：管理窗口生命周期、系统托盘、本地文件访问。
- **React UI**：实现仪表盘（评估、治理事件）、执行时间线、记忆检查器、工作区管理界面。
- **WebSocket Client**：与服务端保持长连接，接收实时事件推送。

### 2.2 Agent 运行时核心
- **Agent Registry & Factory**：管理所有 Agent 类型，支持动态注册和热插拔。
- **AgentBus**：基于 `asyncio.Queue` 构建的发布-订阅消息总线，保证 Agent 间松耦合。
- **Workflow Engine**：自研轻量级 DAG 引擎，支持节点重试、超时、并行执行，以及工作流级别的暂停/恢复/终止。
- **Resource Governor**：包括工具级超时、并发信号量、重试限制器、事件速率限制器、断路器状态机、Agent 进程内存监控。
- **Reflection Engine**：定期执行失败模式挖掘、策略生成、沙箱验证与自动应用。
- **Memory Manager**：统一管理结构化记忆（SQLite）和语义记忆（ChromaDB），提供读/写接口。

### 2.3 工具与基础设施层
- **Tool Execution Layer**：封装各种外部工具调用（CurseForge API、文件系统操作、Minecraft 启动命令等），每个工具调用均受 Resource Governor 约束。
- **Sandbox Workspace Manager**：为每次执行创建独立目录树，利用符号链接实现 Copy-on-Write 共享只读资源，管理生命周期与清理策略。
- **Evaluation Engine**：基于 pandas 和自定义公式引擎，根据执行快照计算多维指标，生成 `EvaluationReport`。
- **External Service Adapters**：统一封装 LLM API、模组仓库 API 等外部服务，带有失败重试和降级逻辑。

### 2.4 数据层
- **SQLite**：存储结构化数据——工作流模板、评估报告、治理事件、配置、用户偏好。**通过 Docker volume 隔离**，路径为 `~/.local/share/modweaver/`。
- **ChromaDB**：存储记忆向量嵌入，支持语义相似度检索。**通过 Docker volume 隔离**。
- **File System**：沙箱工作区目录、日志、快照。**工作区也在 Docker volume 内**，不污染项目目录。

## 3. 运行时进程视图

系统在桌面端运行两个主要进程：

- **Electron 主进程**：管理 UI 渲染进程，并启动后端 Python 子进程。
- **Python 服务进程（FastAPI + Uvicorn）**：承载 Agent 运行时、API 服务和 WebSocket 端点。

进程间通过 WebSocket（或本地 HTTP/WS）通信，Python 进程负责所有核心逻辑，UI 仅呈现状态与发送用户指令。

在 Python 进程内部，各个 Agent 被设计为**独立的 `asyncio.Task`**，它们从 AgentBus 接收消息并处理。工作流引擎作为调度中心，将 Planner 产出的 DAG 拆解为工具调用事件分发给 Executor、Diagnoser 等。

### 典型协作场景：用户请求“安装一个科技整合包并确保能启动”
1. **UI** 发送请求到 `/api/execution/start`。
2. **Sandbox Workspace Manager** 创建 `execution-{id}` 目录。
3. **Planner Agent** 接收 `user_request` 事件，调用 LLM 生成结构化 DAG。
4. **Executor Agent** 按 DAG 顺序执行工具调用（下载模组、修改配置等）。每次工具调用经过 Resource Governor 检查。
5. 若工具失败，**Diagnoser Agent** 分析日志/崩溃报告，发送诊断事件。
6. **Repairer Agent** 根据诊断生成修复动作，重新注入 DAG 执行。
7. 执行过程中，**Memory Agent** 记录所有执行轨迹，**Evaluator Agent** 在关键节点生成评分。
8. 所有事件（包括资源治理事件）通过 WebSocket 推送到 UI 更新时间线。
9. 任务结束，Evaluator 生成最终 `EvaluationReport`，持久化存储。

## 4. 组件详细设计

### 4.1 多 Agent 通信与编排

**AgentBus** 设计：
- 使用 `asyncio.Queue` 为每个 Agent 创建专属消息队列。
- 消息格式采用 **Pydantic 模型**，包含 `agent_id`、`event_type`、`payload`、`correlation_id`、`timestamp`。
- 支持点对点和广播（例如治理事件需通知多个 Agent）。
- 为降低复杂度，初期不实现复杂的消息持久化，但保留日志重建能力。

**Agent 生命周期**：
- 每个 Agent 启动时向 Registry 注册，并创建对应的 Task。
- **【Phase 3 启用 A/B 测试】**：可同时运行不同版本的 Planner（通过不同 `agent_id` 前缀），由实验管理器将用户请求路由至特定 Agent 实例，结果由 Evaluation Engine 比较。Phase 1 只实现单一 Planner，AgentRegistry 预留 `version` 字段。

**DAG 工作流引擎**：
- 工作流定义为 `nodes`（工具调用）和 `edges`（依赖），每个节点带有 `timeout`、`retry_policy`、`fallback_node` 等属性。
- 引擎内建状态机：`PENDING -> RUNNING -> SUCCESS / FAILED / TIMED_OUT`。
- 支持动态修改：Repairer 可插入新的修复节点，或替换当前节点。

### 4.2 资源治理实现

治理策略采用**装饰器模式**切入工具执行函数，统一在 `ResourceGovernor` 中管理。

- **工具超时**：使用 `asyncio.wait_for` 实现。
- **并发限制**：`asyncio.Semaphore(4)` 控制最大并行工具调用。
- **重试限制**：基于异常类型计数，达到阈值后抛出 `RetryLimitExceeded`。
- **断路器**：状态机（关闭→打开）。**【Phase 2 完整实现】** 后续添加半开状态。当前 Phase 1 实现基础版（关闭→打开），连续失败达到阈值后暂停工具/工作流。
- **事件速率限制**：**【Phase 2 降级为】** 计数器+时间窗口。令牌桶算法**【Phase 3 启用】**，文档说明升级路径。
- **Agent 内存监控**：Python 进程内通过 `resource` 模块监控每个 Task 的内存使用（近似），超限则触发 Task 取消并重启 Agent。

### 4.3 反思记忆系统

**记忆类型存储**：
- `FailureMemory`：结构化存储 (模组A, 模组B, 错误类型, 上下文)。
- `ExecutionMemory`：完整执行轨迹快照，含工具调用、状态变化。
- `PersistentMemory`：可执行策略规则文本，验证通过后自动加入。
- `ReflectiveTrajectory`：【新增】模式→策略→验证结果的完整记录，用于回滚。

**反思触发与处理**：
- 由 `ReflectionEngine` 周期性运行（如每完成 10 次执行或手动触发）。
- **注意**：`ReflectionEngine` 的策略生成规则视为**不可变配置**，不直接修改 Planner 的系统提示。策略通过 Memory Agent 的检索上下文注入。
- 流程：
  1. **模式挖掘**：从 FailureMemory 中提取最近 N 条记录，使用 DBSCAN 聚类（特征：模组生态、错误关键词向量、依赖图模式），识别出高频失败簇。
  2. **策略生成**：将失败簇的样本交给 LLM，生成自然语言规则，如”当用户要求包含’Create’模组且使用Forge时，推荐使用NeoForge”。规则同时被转换为结构化条件。
  3. **沙箱验证**：选取历史代表性案例，将新规则注入 Memory Agent 检索结果（而非 Planner 提示词），重放执行并评估指标。
  4. **应用与回溯**：验证通过（成功率提升且未引入新回归）后，自动加入 `PersistentMemory` 并记录到 `ReflectiveTrajectory`。所有反思动作支持回滚。

### 4.4 沙箱工作区隔离

**实现原理**：
- 公共资产（Minecraft 核心 jar、标准库、通用模组）存储于 `/shared/`，按版本与哈希组织。
- 为 `execution-{id}` 创建目录，使用 `os.symlink` 将所需公共资产链接到 `instance/` 下。
- 任何写入操作（如日志、生成配置文件）发生在独立目录内，不会影响公共资产。
- 快照：可复制当前 instance 目录到 `snapshots/` 子目录（轻量级，因部分文件仍是符号链接）。

**生命周期管理**：
- 每次新执行创建新工作区。
- 保留策略：LRU + 最大数量（默认10个）。通过 cron-like 内部调度器检查并清理。
- UI 提供手动清理和”置顶保留”功能。

### 4.5 工作流持久化与断点恢复

**设计目标**：工作流可从中断点恢复，支持在工具调用边界保存状态。

**WorkflowSnapshot 数据结构**：
```python
class WorkflowSnapshot(BaseModel):
    workflow_id: str
    execution_id: str
    completed_nodes: list[str]  # 已完成节点ID列表
    pending_nodes: list[str]    # 待执行节点ID列表
    node_results: dict[str, ToolResult]  # 节点ID → 执行结果
    current_dag_state: dict    # DAG 结构快照（序列化）
    created_at: datetime
    updated_at: datetime
```

**Checkpoint 策略**：
- **触发时机**：每个工具调用完成时（节点状态变为 SUCCESS / FAILED / TIMED_OUT）
- **实现方式**：工具执行完成后立即写入 SQLite（使用 WAL 模式保证一致性）
- **恢复流程**：
  1. 工作流启动时检查是否存在未完成的 `ExecutionSnapshot`
  2. 若存在，加载 `WorkflowSnapshot`，跳过 `completed_nodes` 中的节点
  3. 从 `pending_nodes` 的第一个节点继续执行
  4. 若工作流被外部终止（用户取消），清理 Snapshot

**状态一致性保证**：
- Node 结果在写入 Snapshot 前必须已持久化
- 恢复时校验 DAG 结构版本，若版本变化则拒绝恢复并提示用户
- 恢复后重新创建 Agent Task，而非复用旧 Task（防止状态污染）

## 5. 数据架构

### 5.1 关键数据实体

- **UserRequest**：用户输入的原始文本及解析后意图。
- **WorkflowDefinition**：DAG 结构，包含节点和边，版本化。
- **ExecutionSnapshot**：工作流实例运行时的完整状态序列化快照，用于回放和评估。
- **EvaluationReport**：包含成功率、效率、恢复分数等多维指标。
- **GovernanceEvent**：超时、熔断、限流等事件记录，带时间戳。
- **MemoryEntry**：统一记忆条目，支持结构化字段（JSON）和向量嵌入。
- **ReflectiveTrajectory**：模式→策略→验证结果的完整记录。

### 5.2 存储选型细化
- **SQLite**：存储所有结构化关系数据，使用 WAL 模式提升并发。负责工作流模板、执行记录、评估报告、治理事件等。通过 SQL 查询历史趋势。**运行时数据使用 Docker volume 隔离**，不存放于项目目录。
- **ChromaDB**：嵌入存储，保存记忆条目的语义向量。与 SQLite 通过 `memory_id` 关联，记忆条目在 SQLite 中存元数据，Chroma 中存向量和原始文本。查询时先语义搜索，再回到 SQLite 获取完整信息。**运行时数据使用 Docker volume 隔离**。
- **文件存储**：日志、工作区、快照使用普通文件系统，路径记录在 SQLite 中。工作区目录也在 Docker volume 内。

## 6. 安全架构

- **沙箱限制**：Agent 工具调用被限制在 `execution-{id}` 目录内，通过检查文件路径前缀防止逃逸。
- **工具白名单校验**：**【新增】** 所有 Executor 执行的工具必须在 `allowed_tools.json` 中。Pydantic 验证格式，工具白名单验证语义安全。
- **Subprocess 目录绑定**：**【新增】** 所有 spawn 的 subprocess 必须 `chdir` 到 `execution-{id}` 目录，防止相对路径逃逸。Windows 使用 ACL 强化。
- **LLM 输出校验**：Planner 和 Repairer 生成的结构化输出必须通过 Pydantic 验证，避免注入非法工具调用。
- **API 密钥管理**：LLM 密钥等加密存储在本地 Keychain（Electron 端），不落盘明文。
- **权限控制**：单用户桌面应用，没有多租户，但未来如需共享，可扩展基于能力的安全模型。
- **数据隔离**：**【新增】** 运行时数据（SQLite、ChromaDB、工作区）通过 Docker volume 挂载到 `~/.local/share/modweaver/`，不污染项目目录。

## 7. 部署与打包

- **开发环境**：Python 3.12+ + Node.js 20+，依赖管理用 **uv**（替代 Poetry）和 pnpm。
- **数据服务**：运行时数据使用 **Docker** 隔离（SQLite、ChromaDB、工作区通过 Docker volume 管理）。
- **打包分发**：
  - 前端通过 Vite 构建，由 electron-builder 打包为 Asar。
  - 后端 Python 使用 PyInstaller 打包成独立可执行文件，包含 Python 解释器及所有依赖。
  - 最终交付物为单文件安装包（Windows NSIS / macOS DMG / Linux AppImage）。

## 8. 关键技术决策记录

| 决策点     | 选择                          | 理由                                                         | 备选方案                                        |
| ---------- | ----------------------------- | ------------------------------------------------------------ | ----------------------------------------------- |
| Agent 通信 | 自研 AgentBus + asyncio.Queue | 零依赖、高性能，完全控制消息流与反压，适合中等数量 Agent。   | NATS / Redis Pub/Sub（过重，不适合嵌入式桌面）  |
| 工作流引擎 | 自研 DAG 引擎                 | 需求高度定制（动态修改节点、重试/超时/熔断集成），现有框架（Temporal/Airflow）过重。 | 无，但可借鉴 Prefect 模式                       |
| 向量数据库 | ChromaDB                      | 嵌入式运行，无需额外服务，支持持久化，API 简洁。             | FAISS（需自行封装持久化），Qdrant（需额外进程） |
| 沙箱隔离   | 文件系统符号链接              | 轻量、跨平台（Windows 也支持），无虚拟化开销，满足隔离需求。 | Docker 容器（太重，启动慢，桌面环境不易分发）   |
| 评估计算   | pandas + 自定义公式           | 灵活处理多指标聚合，pandas 生态成熟，未来可扩展复杂统计分析。 | 纯 Python 统计（工作量大）                      |
| 依赖管理   | uv                            | 快速、确定性强的 Python 包管理，替代 Poetry。                 | Poetry、pip                                     |
| 数据隔离   | Docker volume                 | 运行时数据隔离，保护项目目录不被污染。                        | 直接存放在项目目录（不推荐）                    |

## 9. 关键质量属性满足

- **性能**：工作区创建通过符号链接实现 <500ms；Agent 通信异步非阻塞。
- **可扩展性**：Agent 注册机制使新 Agent 无需修改调度器；评估指标可通过配置文件添加自定义公式。
- **可观测性**：所有运行时事件流向治理时间线；评估仪表盘展示多维度图表；记忆检查器支持检索和追踪反思链条。
- **可靠性**：断路器与重试限制确保外部依赖故障不扩散；工作流状态可持久化并从中断处恢复；隔离环境保证重放一致性。

## 10. 架构演进与开源准备

当前架构为桌面单体应用，但核心 Agent 运行时与 UI 通过明确 API（WebSocket/REST）解耦，未来可轻松拆分为后端服务，支持远程执行。沙箱与资源治理的抽象也使得引入容器化隔离成为可能。代码模块化、文档完善后，以 MIT 协议开源，社区可贡献新的 Agent 类型、治理策略或评估指标。
