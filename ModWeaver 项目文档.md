## 一、项目概述

### 1.1 定位

ModWeaver 是一个以 **Agent 技术学习**为目的的练习项目，以 **Minecraft 整合包搭建**为具体练习场景。核心目标不是交付产品，而是通过实践理解 Agent 编排、协作、记忆、评估等核心技术。

### 1.2 设计原则

1. **不重复造轮子**——优先使用 LangGraph、ChromaDB、tenacity 等现有框架和库
2. **渐进式实现**——从单 Agent 原型起步，逐步增加复杂度
3. **可感知的进展**——每个 Stage 有明确产出，不做无法运行的架构设计
4. **文档服务于代码**——文档先于代码，但随代码迭代同步更新

## 二、核心概念

### 2.1 Agent 角色

| Agent       | 职责                       | 输入             | 输出                     |
| :---------- | :------------------------- | :--------------- | :----------------------- |
| **Planner** | 解析需求，生成任务列表     | 用户文本         | 结构化任务定义           |
| **Executor** | 按任务调用工具，处理执行   | 任务节点         | 执行结果、状态变更       |
| **Diagnoser** | 分析崩溃日志，输出根因假设 | 崩溃日志、上下文 | 根因分类、可疑模组列表   |

三个阶段逐步启用，Stage 1 仅 Planner + Executor，Stage 2 加入 Diagnoser。

### 2.2 典型流程

```
用户输入 → Planner 生成计划 → Executor 执行 → (如果崩溃) Diagnoser 诊断 → 修复建议
```

### 2.3 技术架构

```
┌────────────────────────────────────────────────┐
│                Agent Runtime                      │
│  ┌──────────────────────────────────────┐       │
│  │        LangGraph StateGraph           │       │
│  │  Planner → Executor → Diagnoser      │       │
│  └──────────────────────────────────────┘       │
│  ┌──────────────────────────────────────┐       │
│  │     Memory System (ChromaDB)         │       │
│  └──────────────────────────────────────┘       │
│  ┌──────────────────────────────────────┐       │
│  │    Tool Execution Layer (Python)     │       │
│  └──────────────────────────────────────┘       │
└────────────────────────────────────────────────┘
```

### 2.4 技术选型

| 组件           | 方案                                  | 说明                         |
| :------------- | ------------------------------------- | ---------------------------- |
| Agent 框架     | **LangGraph**                         | 内建状态图、检查点、消息路由 |
| 数据库         | **SQLite**（结构化）+ **ChromaDB**（语义） | 开发用 Docker Compose 管理    |
| 重试/限流      | **tenacity** + **asyncio**            | 不再自研限流器               |
| 数据计算       | **pandas**                            | 保留，仅按需使用             |
| LLM 框架       | **LangChain**（与 LangGraph 一致）     | 统一生态                     |
| 数据隔离       | Docker Compose（开发环境）            | 部署时可切换到本地 SQLite    |

## 三、路线图（Stage 制）

### Stage 0：原型验证

**目标**：确认 LLM 能有效诊断 Minecraft 崩溃

- 单个 Python 脚本，一个 LLM 调用，输入硬编码的 Minecraft 崩溃日志
- 验证：根因判断准确度、模组冲突识别能力、常见错误理解
- 产出：README 中记录 5-10 个真实测试案例的诊断结果
- **Done 定义**：能正确区分 "依赖缺失" "版本冲突" "生态不兼容（Forge vs Fabric）" 三类问题

### Stage 1：双 Agent 端到端链路

**目标**：用 LangGraph 实现一个可运行的双 Agent 协作流程

- Planner：解析用户需求（如 "安装 Create 模组到 1.20.1 Forge"），生成结构化步骤
- Executor：按步骤调用工具（查询模组信息、分析兼容性），返回结果
- LangGraph StateGraph 管理状态流转
- Docker Compose 启动 SQLite + ChromaDB
- **Done 定义**：用户输入一句需求 → 能输出 "可安装" 或 "存在以下兼容问题" 的分析结果

### Stage 2：诊断闭环

**目标**：加入 Diagnoser，形成完整 "需求 → 分析 → 诊断 → 修复建议" 链路

- Diagnoser：输入崩溃日志 → 输出根因假设（含置信度）和可疑模组列表
- Executor 调用 Minecraft 启动工具模拟崩溃场景（或在真实日志上回放）
- **Done 定义**：能复现 Stage 0 中的测试案例，且诊断结果与人工判断一致

### Stage 3：记忆系统

**目标**：让 Agent 能利用历史经验

- ChromaDB 存储失败案例的语义向量
- Memory Agent（或 LangGraph 内建 memory）在诊断时检索历史相似案例
- 同一类问题第二次出现时直接给出参考策略
- **Done 定义**：同一崩溃场景第二次处理时，诊断时间缩短 50% 以上

### Stage 4：评估与保护

**目标**：让运行时具备基本的可观测性和自我保护

- 收集每次执行的耗时、token 消耗、工具调用成功率
- tenacity 控制重试策略，asyncio 设置调用超时
- 简单的评估报告输出（pandas 汇总）
- **Done 定义**：Agent 在 LLM 调用异常或工具超时时能优雅降级并输出告警

## 四、待定方向（后续探索）

以下方向在完成 Stage 4 后按兴趣选择实现，当前不投入设计：

- **反思记忆**：从失败中自动提取模式，生成新策略（需验证聚类分析的有效性）
- **Agent 协商/投票**：多个 Agent 对同一决策问题投票（提案、投票规则、共识机制、仲裁）
- **Electron UI**：目前先 CLI 界面，后续根据需要决定是否做桌面壳
- **扩展 Agent 角色**：Repairer（自动修复）、Memory（专职检索）、Evaluator（专职评估）
