## 一、产品战略定位

### 1.1 产品定位

ModWeaver V2.5 是一个**多Agent协作、可自我进化、带资源治理与沙箱隔离的Minecraft运行时编排平台**。它不仅是模组管理工具，更是面向AI Agent Runtime Engineering的真实复杂环境实验场。

### 1.2 核心特征

- **多Agent运行时**：专业化分工的Agent群体（Planner、Executor、Diagnoser、Repairer、Memory、Evaluator）
- **评估引擎**：量化Agent执行质量，驱动决策与优化
- **资源治理**：防止运行时失控，保障稳定性
- **反思记忆**：从失败中提取模式，自动调整策略，实现自我改进
- **沙箱隔离**：每次执行独立工作区，确保可重现与彻底清理

### 1.3 升级对照

| 能力维度   | V2.0（单体Agent运行时）   | V2.5（多Agent+自我进化）                              |
| :--------- | :------------------------ | :---------------------------------------------------- |
| Agent架构  | 一个全能Agent调用所有工具 | 多Agent协作：Planner、Executor、Diagnoser、Repairer等 |
| 质量评估   | 无/主观                   | 内置评估引擎，量化成功率、效率、恢复能力              |
| 容错与治理 | 基础重试                  | 熔断器、全局超时、事件限流、资源限制                  |
| 记忆系统   | 存储事实（成功/失败组合） | 反思记忆：自动提取模式并调整策略                      |
| 执行隔离   | 共享临时目录              | 每次执行独立工作区（Copy‑on‑Write）                   |

## 二、项目范围

### 2.1 核心功能模块

- **多Agent运行时**：实现Planner、Executor、Diagnoser、Repairer、Memory、Evaluator六个专职Agent，支持独立部署与替换。
- **评估引擎**：自动生成每次执行的量化报告，支持A/B测试不同Agent配置。
- **资源治理层**：包含超时控制、重试限制、熔断器、并发限制、事件速率限制及内存限制。
- **反思记忆系统**：基于失败记录的聚类分析，自动生成策略并验证后应用。
- **沙箱工作区隔离**：每次执行映射到独立目录，使用符号链接优化存储，支持完整清理与回放。

### 2.2 明确不包含（当前版本）

- mDNS服务发现、防火墙自动配置、客户端‑服务端差分同步。
- Java Agent级深度诊断模组。
- 云端协作与多租户共享执行环境。
- 可视化拖拽式工作流编辑器。

## 四、功能需求

### 4.1 多Agent运行时

**描述**：将单体Agent拆分为多个专职Agent，通过异步消息协作完成复杂任务。

**Agent定义**：

| Agent名称     | 职责                                      | 输入                 | 输出                         |
| :------------ | :---------------------------------------- | :------------------- | :--------------------------- |
| **Planner**   | 解析自然语言需求，生成可执行工作流（DAG） | 用户文本             | 结构化Workflow定义           |
| **Executor**  | 按计划调用工具，处理确定性执行            | Workflow节点         | 工具执行结果、状态变更       |
| **Diagnoser** | 分析崩溃/失败，输出根因假设及可疑模组     | 崩溃日志、执行上下文 | 根因分类、可疑列表           |
| **Repairer**  | 基于诊断结果生成并尝试修复方案            | 诊断报告             | 修复动作（降级、排除、配置） |
| **Memory**    | 管理所有记忆的读写与检索                  | 查询请求             | 记忆条目                     |
| **Evaluator** | 对执行结果进行质量评估                    | Execution Snapshot   | 评估报告（分数、指标）       |

**技术约束**：
- 每个Agent运行在独立事件循环任务中，通过`AgentBus`传递消息。
- Agent间不共享内存，仅通过事件通信。
- 支持Agent的热插拔与A/B测试。

### 4.2 评估引擎

**描述**：对每次执行或单个Agent行为进行量化评分，用于比较和优化。

**核心指标**：

| 维度     | 指标                             | 计算方式                             | 目标范围 |
| :------- | :------------------------------- | :----------------------------------- | :------- |
| 成功率   | `task_success_rate`              | 最终达成用户目标（如游戏启动）的比例 | ≥0.7     |
| 效率     | `tool_efficiency`                | 有效工具调用次数 / 总工具调用次数    | ≥0.6     |
| 恢复能力 | `recovery_score`                 | 成功恢复次数 / 崩溃总次数            | ≥0.5     |
| 资源消耗 | `total_time_ms`, `token_usage`   | 总耗时、LLM token消耗                | 越低越好 |
| 稳定性   | `retry_count`, `workflow_aborts` | 重试次数、工作流异常终止次数         | ≤3       |

**功能要求**：
- 每次执行自动生成`EvaluationReport`并存入`ExecutionMemory`。
- 支持对比两次执行（如不同Planner配置）并生成差异报告。
- 评估函数可自定义（Python脚本）。
- UI提供历史执行趋势图、Agent效率热力图。

### 4.3 资源治理层

**描述**：提供运行时防护机制，防止资源耗尽、死锁或无限循环。

**治理策略表**：

| 资源/风险        | 治理手段            | 默认限制      | 超限动作                     |
| :--------------- | :------------------ | :------------ | :--------------------------- |
| 单工具调用时长   | 工具级超时          | 30秒          | 取消调用，标记`TOOL_TIMEOUT` |
| 单工作流总时长   | 工作流超时          | 10分钟        | 终止工作流，触发恢复         |
| 工具重试次数     | 重试限制            | 3次           | 超过后标记失败，不再重试     |
| 并发工具数       | 并发限制            | 4个           | 排队等待                     |
| 连续失败（熔断） | 断路器              | 5次连续失败   | 暂停该工具/工作流1分钟       |
| 事件风暴         | 事件速率限制        | 1000事件/分钟 | 降级为批量聚合事件           |
| Agent内存膨胀    | 内存限制（每Agent） | 512 MB        | 强制终止Agent进程            |

**功能要求**：
- 运行时配置可修改（`runtime_settings.json`或UI）。
- 熔断器状态变化必须在`Execution Timeline`上可见。
- 资源紧张时自动暂停低优先级任务。

### 4.4 反思记忆系统

**描述**：在失败记忆之上建立模式识别与策略自动调整机制，实现Agent的自我进化。

**能力对比**：

| 记忆类型              | 存储内容                  | 查询方式    | 能否改变行为 |
| :-------------------- | :------------------------ | :---------- | :----------- |
| `FailureMemory`       | (模组A, 模组B) → 崩溃类型 | 精确匹配    | 仅用于诊断   |
| `ExecutionMemory`     | 执行轨迹快照              | 时间/ID检索 | 仅用于回放   |
| `PersistentMemory`     | 可执行策略规则文本        | 语义相似度  | 自动修改行为 |
| `ReflectiveTrajectory` | 模式→策略→验证完整记录    | 回滚检索    | 支持回滚     |

**功能要求**：
- **模式挖掘**：定期对失败记录进行聚类（如DBSCAN），识别高频失败类型（例如”Fabric API + Forge专属模组失败率>80%”）。
- **策略生成**：调用LLM根据失败模式生成新规则（如：`IF 用户需求含”科技” AND 模组列表含”Forge专属” THEN 推荐NeoForge`）。
- **策略验证与应用**：新策略先在沙箱中测试，若提升成功率则自动加入`PersistentMemory`，并记录到`ReflectiveTrajectory`用于回滚。**策略通过Memory Agent检索上下文注入，不直接修改Planner系统提示**。
- **反思轨迹记录**：每次反思（模式→策略→效果）作为元学习轨迹，支持回放。

### 4.5 沙箱工作区隔离

**描述**：为每次执行分配完全独立的文件系统工作区，避免污染，便于重现与调试。

**目录结构**：
```
/workspaces/
    execution-<id>/
        instance/          # Minecraft实例（mods、config、saves）
        logs/              # 本次执行所有日志
        snapshots/         # 中间状态快照
        metadata.json      # 执行ID、开始时间、Git版本等
    execution-<id+1>/
        ...
```

**功能要求**：
- 每次新执行或Replay自动创建独立工作区。
- 只读公共资产（如相同版本的Minecraft jar、相同模组文件）使用符号链接（Copy‑on‑Write）以节省空间。
- 执行完成后按策略保留最近N个工作区（默认10个），其余可手动或自动清理。
- Replay时直接加载对应工作区快照，无需重新下载。
- UI提供“清理所有残留工作区”按钮。

## 五、非功能需求

| 类别         | 要求                                                         | 目标值/约束      |
| :----------- | :----------------------------------------------------------- | :--------------- |
| **性能**     | 工作区创建（含符号链接设置）时间                             | < 500 ms         |
| **可靠性**   | 断路器触发后，运行时必须记录事件并支持恢复                   | 必须             |
| **安全性**   | Agent工具不得访问工作区外的文件系统路径                      | 必须（沙箱限制） |

| **可观测性** | 所有资源治理事件（超时、熔断、限流）必须出现在`Execution Timeline` | 必须             |
| **数据隔离** | 运行时数据（数据库、日志、工作区）必须与项目目录隔离           | 必须（Docker volume） |
| **可维护性** | 代码注释覆盖率 ≥ 60%，提供架构文档与开发者指南               | 目标             |

## 九、风险与应对

| 风险                        | 概率 | 影响 | 应对措施                                                     |
| :-------------------------- | :--- | :--- | :----------------------------------------------------------- |
| Agent间通信复杂化           | 中   | 中   | 严格定义事件Schema（Pydantic校验）；初期限制Agent数量≤4      |
| 反射策略可能引入错误行为    | 高   | 高   | 新策略必须在沙箱工作区中验证成功后自动应用；保留策略回滚机制（ReflectiveTrajectory） |
| 工作区隔离导致磁盘占用膨胀  | 中   | 低   | 实施LRU清理策略（默认保留最近10个），提供一键清理工具；支持将工作区目录挂载到外部存储 |
| 评估指标被“作弊”优化        | 低   | 中   | 使用多维度复合评分，禁止单一指标导向；公开评估基准集与测试用例 |
| 工具API（CurseForge等）变更 | 中   | 高   | 多源备份（Modrinth等），本地缓存机制；提供离线模式降级       |

## 十、总结

ModWeaver V2.5 将项目从“单体Agent工具”演进为**完整的多Agent运行时编排平台**，具备了协作分工、量化评估、资源治理、自我反思和强隔离等企业级能力。通过在Minecraft这一复杂真实环境中实践这些前沿概念，ModWeaver将成为开源社区中Agent Engineering的标杆参考实现。

**核心价值**：
> 从“一个Agent干所有事”到“一群Agent各司其职、相互协作，且能自我批评、自我改进”，为构建高可靠的AI Agent系统提供可复制的实践范本。
